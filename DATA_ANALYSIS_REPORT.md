# NVIDIA Nemotron Model Reasoning Challenge データ分析レポート

## 対象

このレポートは、ローカルにある以下の資料を再分析してまとめたものです。

- `README.md`
- `data/train.csv`
- `data/test.csv`

目的は、データセットの実際の構造、`prompt` / `answer` の性質、コンテキスト長の観点、そして実務的なモデリング上の示唆を整理することです。

## README から読み取れる前提

`README.md` から確認できる重要な前提は次のとおりです。

- 最終提出物は `Nemotron-3-Nano-30B` 向けの **LoRA adapter**。
- 評価指標は **Accuracy**。
- 生成時は、最終回答をできれば `\boxed{}` に入れることが期待されている。
- 抽出ロジックは `\boxed{}` を優先するが、入っていない場合はヒューリスティックに最終答えを拾う。
- README に記載された推論条件は以下。
  - `max_model_len = 8192`
  - `max_tokens = 7680`
  - `temperature = 0.0`
  - `top_p = 1.0`
- `test.csv` はあくまで **サンプル用のテストセット**であり、本番採点時には **数百問の hidden test** に差し替わる。

README では「bit manipulation や algebraic equations を含む論理推論問題」と広く説明されていますが、実データを見ると、実態はかなり強くテンプレート化された合成タスク群です。

## ファイル単位の要約

| ファイル | 行数 | カラム | 役割 |
| --- | ---: | --- | --- |
| `data/train.csv` | 9,500 | `id`, `prompt`, `answer` | 学習用の主データ |
| `data/test.csv` | 3 | `id`, `prompt` | 提出確認用のサンプル |

### 重要な発見: `test.csv` は検証用データではない

`data/test.csv` の 3 件は、`id` でも `prompt` でも **すべて `train.csv` と完全一致**していました。

つまり可視の `test.csv` は次の用途には使えます。

- 推論コードの入出力確認
- submission 生成フローの確認
- 回答抽出フォーマットの確認

ただし、**性能評価には使えません**。Leaderboard に近い見積もりを出すには、`train.csv` から自前で validation split を作る必要があります。

## 全体統計

### Prompt 長

プロンプトはかなり短く、しかも形式が揃っています。

| 指標 | 文字数 | 単語数 | 近似 token 数* |
| --- | ---: | ---: | ---: |
| Min | 177 | 32 | 46 |
| Median | 281 | 46 | 69 |
| Mean | 301.52 | 50.04 | 72.83 |
| Max | 510 | 78 | 110 |

\* 近似 token 数は、この環境に対象モデル用 tokenizer がなかったため、単語と記号の分割で概算した値です。

### Answer 長

| 指標 | 値 |
| --- | ---: |
| 最小文字数 | 1 |
| 中央値 | 5 |
| 平均文字数 | 8.39 |
| 最大文字数 | 39 |
| ユニークな answer 文字列数 | 5,921 |

### コンテキスト長の観点

README の `max_model_len = 8192` と比べると、各問題の `prompt` は非常に短いです。

- 中央値は **約 69 token**
- 最大でも **約 110 token**

したがって、1 問ごとの推論ではコンテキスト長はほぼボトルネックになりません。実務上は次のことが言えます。

- 強めの system prompt を入れる余裕がある
- few-shot 例を足す余裕がある
- 回答フォーマット指示を明示しても長さ面で困りにくい
- 長文読解よりも、テンプレ判定と最終出力の安定性が重要

## データセット構成: 6 種類のほぼ均衡なテンプレート

README の説明は抽象的ですが、実際の `train.csv` は **6 種類の強くテンプレート化されたタスク**にほぼ分かれます。しかも件数はかなり均等です。

| タスク種別 | 件数 | 構成比 | prompt 中央文字数 | 近似 token 中央値 | 典型的な answer |
| --- | ---: | ---: | ---: | ---: | --- |
| 8bit binary の bit 操作 | 1,602 | 16.86% | 489 | 97 | 8 文字の 2 進数 |
| 重力定数が変わった落下計算 | 1,597 | 16.81% | 319 | 97 | 小数の数値 |
| 単位変換 | 1,594 | 16.78% | 223 | 61 | 小数の数値 |
| テキスト復号 | 1,576 | 16.59% | 370 | 70 | 複数語の英語句 |
| ローマ数字変換 | 1,576 | 16.59% | 219 | 50 | ローマ数字 |
| 記号/式の変換 | 1,555 | 16.37% | 195 | 61 | 短い記号列または整数 |

### 1. Bit manipulation

- 「秘密の bit 操作ルール」が 8bit binary に適用されるという設定
- いくつかの input -> output 例が与えられる
- 新しい 8bit 入力の出力を求める
- answer は常に **8 文字の 2 進文字列**

この系列は prompt が最も長いです。理由は、例示数が多く、しかも shift / rotation / XOR / AND / OR / NOT といった操作候補の説明が毎回入るためです。

### 2. 重力定数が変わった落下計算

- `(t, distance)` の観測例がいくつか与えられる
- `d = 0.5 * g * t^2` が明示される
- 新しい `t` に対する落下距離を求める
- answer は数値

数値フォーマットは比較的安定しています。

- `1,440` 件が **小数第 2 位**
- `157` 件が **小数第 1 位**

### 3. 単位変換

- 変換例が数件与えられる
- 新しい入力値の変換結果を答える
- answer は **常に小数第 2 位の数値**

この系列は短く、構造もかなり素直です。

### 4. テキスト復号

- 暗号文 -> 平文の例が複数与えられる
- 新しい暗号文を復号する
- answer は英語の複数語フレーズ

この系列は answer が最も長い傾向にあります。

- answer 長の中央値は **約 25 文字**
- 最大は **39 文字**

### 5. ローマ数字変換

- 10 進数 -> ローマ数字の例が与えられる
- 新しい数値をローマ数字に変換する
- answer は常に **ローマ数字**

この系列は最も短く、構造も非常にクリーンです。

### 6. 記号 / 式変換

- 記号を多く含む短い式の変換例が与えられる
- 新しい式の変換結果を求める
- answer は短い記号列か整数

この系列は出力型が最もばらつきます。

- `867` 件が **記号列中心の出力**
- `688` 件が **整数出力**

prompt 自体は短いものの、answer 型のばらつきという意味では最もノイジーです。

## Answer 型の分布

全 9,500 件を answer 形式で見ると、分布は次のようになります。

| answer 型 | 件数 | 構成比 |
| --- | ---: | ---: |
| 数値 | 3,879 | 40.83% |
| 8bit binary 文字列 | 1,602 | 16.86% |
| 複数語テキスト | 1,576 | 16.59% |
| ローマ数字 | 1,576 | 16.59% |
| 記号系 / その他短文字列 | 867 | 9.13% |

重要なのは、この benchmark が **単一の出力モードではない**ことです。競争力のあるシステムには、少なくとも次の出力を安定して出せることが求められます。

- 数値
- 2 進文字列
- ローマ数字
- 複数語テキスト
- 短い記号列

## 重複と多様性

### Prompt の重複

- `train.csv` 内に **重複 prompt は 0 件**

テンプレートは固定的ですが、prompt がそのまま重複しているわけではありません。つまり「同じ問題文の焼き直し」で構成されているデータではないです。

### Answer の重複

- **936 種類の answer 文字列**が 2 回以上出現

重複しやすい answer は、たとえば次のような低エントロピーの出力空間に集中しています。

- `00000000` のような 8bit binary
- ローマ数字
- 短い数値

したがって answer 重複は、prompt 重複というより **出力空間の狭さ**に起因していると考えるのが自然です。

## 構造的な解釈

このローカル学習データは、広い意味での自然言語推論コーパスというより、**合成的な rule induction タスクの混合データ**として見るほうが実態に近いです。

特に重要なのは次の点です。

1. **テンプレ判定が容易**  
   prompt の最初の 1 文だけで、ほぼタスク種別が分かります。

2. **クラスバランスが良い**  
   6 系列がかなり均等で、1 系列だけに強く偏っていません。

3. **長い推論文脈は不要**  
   多くの問題は、少数の例から変換規則を推定するタスクであり、長文理解や長距離依存は中心ではありません。

4. **最終出力の整形が重要**  
   評価は最終文字列抽出に強く依存するため、正解そのものに加えて出力形式の安定性がかなり重要です。

5. **可視 `test.csv` は誤解を招きやすい**  
   train と重複しているため、これで性能を見積もると過大評価になります。

## モデリング上の示唆

### 1. 汎用一本槍より、テンプレ意識のある戦略が合う

6 系列は見分けやすいため、次のような発想が自然です。

- まずタスク種別を見抜く
- 系列ごとに answer 形式を強く安定化する
- 系列ごとの exemplars や reasoning trace を使い分ける

最終提出が 1 つの LoRA adapter であっても、学習や prompt 設計の段階で「潜在タスク別」に最適化する価値があります。

### 2. Answer formatting は主要課題

評価側は `\boxed{}` を優先するので、推論時や学習時には少なくとも次を徹底したいです。

- 最終答えを常に canonical form で出す
- 数値問題で余計な数値を後ろに出さない
- ローマ数字や binary、記号列で表記ぶれを起こさない

### 3. Few-shot は十分に載る

prompt 自体が短いので、`8192` token の範囲内で次を追加しても余裕があります。

- system 指示
- compact な few-shot 例
- 系列別 exemplar
- answer 形式の再指示

つまり、この問題では「長文コンテキスト能力」よりも「短い規則誘導を安定して解く prompt 設計」のほうが効きやすい可能性があります。

### 4. Validation は `train.csv` から切るべき

少なくとも次のどれかを自前で作るべきです。

- ランダム split
- テンプレート層化 split
- answer 形式や語彙パターンを意識した harder split

可視の `test.csv` を評価用に使うのは危険です。

## 次にやる価値が高い分析

モデリングに進むなら、次の分析が特に有効です。

1. `train.csv` から **template-stratified validation split** を作る
2. 各テンプレートごとに baseline accuracy を測る
3. few-shot あり / なし、answer format 指示あり / なしで差を比較する
4. 各系列の規則がどれくらい deterministic かを調べる
5. `\boxed{}` を強制したときの extraction 安定性を検証する

## 結論

このローカルデータセットは、**短コンテキスト・合成的・均衡・強テンプレート型**の benchmark です。広い意味での汎用 reasoning よりも、**テンプレ判定、規則推定、そして最終 answer の厳密な整形**が勝負になりやすい構造です。したがって、長文能力そのものより、

- テンプレートを意識した prompting / training
- answer format の厳密化
- `train.csv` 由来の適切な validation

のほうが、実際の改善に直結しやすいと考えられます。
