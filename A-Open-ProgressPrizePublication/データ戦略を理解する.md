## 1. 全体戦略の思想

README に明記された **設計目標** は次の通りです。

| 目標 | 内容 |
| --- | --- |
| **Deterministic** | 評価の temperature=0.0 を前提。最も確率の高いトークンを正しく出せばよい。**min logprob を最大化**することを学習指標にする |
| **Simple** | CoT は「各トークンをできるだけ少ない源泉から導出できる」ように設計する。除算は加算・減算に分解するなど |
| **Coverage** | レアな操作を十分に学習させる。例：Cipher の wonderland word を最初でマッチしたところで止めず、全 77 語を照合してトークン網羅率を上げる |
| **Within length limit** | 推論トークン数 7680 以内を守る |
| **Tokenization awareness** | モデルの tokenizer 弱点を避けた CoT を書く |
| **Generalizable** | 答えの暗記ではなく手順を学習させる |

**非目標**:

- Diversity（多様な出力は不要。temperature=0 なので1つのコレクトな解法で十分）
- Conciseness（CoT が長くても良い）
- Verification（CoT 中のエラー検出・リカバリは実装しない）

## 4. 競技問題 CoT データ（reasoning/txt）

### 生データの所在

```
nemotron/reasoning/<problem_id>.txt
```

1 ファイル = 1 競技問題の CoT テキスト全文（`\boxed{answer}` で終わる）。  
例: `reasoning/00d9f682.txt` は Numeral 問題「100 を Roman Numeral に変換せよ」の完全な CoT。

```
We need to determine the conversion rule from the examples:
I will put my final answer inside \boxed{}.
  4 -> IV
  42 -> XLII
  ...
Reference table (1-100):
  1 = I
  2 = II
  ...
Converting 100:
  100 >= 100 -> C, remainder 0

Result: C -> C

The answer in \boxed{-} is \boxed{C}
```

### 対応する問題定義

```
nemotron/problems/<problem_id>.jsonl
```

```json
{
  "id": "00d9f682",
  "category": "numeral",
  "prompt": "In Alice's Wonderland, numbers are secretly converted into a different numeral system...",
  "answer": "C",
  "examples": [{"input_value": "4", "output_value": "IV"}, ...],
  "question": "100"
}
```

### corpus.py での処理（L222-297）

```python
# prompt + reasoning を結合して学習例を作る
completion_text = f"{reasoning_text}\n</think>\n\\boxed{{{reasoning_answer}}}<|im_end|>"
completion_ids = tokenizer.encode(completion_text, ...).ids
prompt_ids = tokenize_prompt(prompts[problem_id], chat_tokenizer)
all_tokens = prompt_ids + completion_ids
mask = [0] * len(prompt_ids) + [1] * len(completion_ids)
# prompt はマスク（損失計算外）、completion のみ損失計算対象
```

### カテゴリごとの CoT 生成器

`reasoning.py` に登録された generator 対応表:

```python
GENERATORS = {
    "gravity":                  reasoning_gravity,
    "unit_conversion":          reasoning_unit_conversion,
    "cipher":                   reasoning_cipher,
    "bit_manipulation":         reasoning_bit_manipulation,
    "numeral":                  reasoning_numeral,
    "equation_numeric_deduce":  reasoning_equation_numeric,
    "equation_numeric_guess":   reasoning_equation_numeric,
    "cryptarithm_deduce":       reasoning_cryptarithm,
    "cryptarithm_guess":        reasoning_cryptarithm,
}
```

各カテゴリに対してルールベースの決定論的 CoT を生成する Python 関数が存在する。  
学習は「CoT を暗記させる」のではなく「この手順を token レベルで安定再生させる」が目的。

## 6. 人工データと競技カテゴリの対応関係

| 人工データ | 主対象の競技カテゴリ | 信頼度 | 根拠 |
| --- | --- | --- | --- |
| **matching** | **bit_manipulation** | ★★★（非常に高い） | `augmenters/matching.py` L19 に `REASONING_DIR = .../reasoning` があり、`_extract_all_sections()` が bit_manipulation の CoT から `Matching output / Left / Right` を抽出。出力形式が完全に一致 |
| **spelling** | **cipher** | ★★☆（高い） | README の Cipher 節直後（L80）に「Nemotron is quite bad at spelling」と明記。`reasoners/cipher.py` も `Breaking down into characters:` で文字分解を多用。`augmenters/spelling.py` は単語を `-s-e-x-` 形式に分解する訓練 |
| **concatenation** | **cryptarithm_deduce, cryptarithm_guess** | ★★★（非常に高い） | README Cryptarithm 節（L111）に「問題を concatenation / reverse concatenation 判定に還元した」と明記。`reasoners/cryptarithm.py` 冒頭に「Currently handles concatenation operators only」。CoT も `【a】【b】...` 型の括弧記号列を扱う |
| **splitting** | **cryptarithm_deduce, cryptarithm_guess** | ★★★（非常に高い） | README L117 に「quite bad at splitting and concatenating symbols」と Cryptarithm 文脈で明記。`reasoners/cryptarithm.py` は個別 bracket 形式 `【a】【b】【c】` を多用 |
| **lstrip** | cryptarithm 系（推定） | ★☆☆（中） | bracketed symbol 形式で `concatenation/splitting` と同じ記号世界だが、直接抽出や README 明示記述はない |

### なぜ「競技 CoT を増やす」のではなく「補助データを大量投入」するのか

1. **この戦略の本質**: モデルに「賢く考えさせる」のではなく、**決定論的 CoT を token レベルで崩さず再生させる**こと
2. **ボトルネック**: CoT の失敗原因は推論力不足ではなく、**低レベル token 操作（文字分解・括弧操作・bit 照合）の不安定さ**
3. **対策**: 競技 CoT の難所を局所分解した補助データで大量練習 → min logprob が上がる

特に `matching` は完全な別タスクではなく、**bit_manipulation の CoT の一部を切り出して別問題化した**ものです。「競技と無関係な大量人工データ」ではなく、**競技 CoT の難所を分解して重点学習する設計**です。

### 注意: concatenation/splitting/lstrip のコメントと実装の差

これらの augmenter コメントには「cryptarithm problems の symbols を使う」と書いてあるが、**実コードは競技問題ファイルを読まず固定 SYMBOLS リストを使っている**。したがって「特定の競技問題に1対1対応」ではなく、「cryptarithm で必要な低レベル記号操作を一般化して練習させる」設計です。

## 7. Duplication（同一例の再投入）

### 目的

Data augmentation ではなく、**oversampling**（重要問題を重み付け）。

### 実装（corpus.py L349-387）

```python
def _duplicate_entry(e: CorpusEntry, suffix: str) -> CorpusEntry:
    # tokens・mask・answer をそのままコピー（内容は全く同じ）
    dup = CorpusEntry(problem_id=f"{e.problem_id}{suffix}", ...)
    return dup

# 1. priority IDs は "-p0" サフィックスで1コピー追加
priority_dups = [_duplicate_entry(e, "-p0") for e in entries if e.problem_id in priority_ids]

# 2. カテゴリ別dup（cryptarithm_deduce は DUPLICATE_COUNTS=1 → "-d0" で1コピー追加）
cat_dups = [
    _duplicate_entry(e, f"-d{i}")
    for e in entries
    if "-p0" not in e.problem_id
    for i in range(DUPLICATE_COUNTS.get(e.category, 0))
]
```

### Duplication の実例

```json
{"problem_id":"0031df9c","category":"bit_manipulation","answer":"00110100","included":true}
{"problem_id":"0031df9c-p0","category":"bit_manipulation","answer":"00110100","included":true}
```

`-p0` は別問題ではなく、**完全に同一の token 列の再投入**。

### Priority IDs の規模

`investigators/priority_problem_ids.txt` に **1,249 件**（主に bit_manipulation 問題）が記録されており、これらはすべて `-p0` コピーが追加されている。

### カテゴリ別 Duplication 設定（corpus.py L60-62）

```python
DUPLICATE_COUNTS: dict[str, int] = {
    "cryptarithm_deduce": 1,   # 全件 1 コピー追加（54 → 125）
}
```

cryptarithm_deduce は学習可能件数が 54 件と極めて少ないため、全件複製で補強している。

## 8. 学習時のデータ混合・処理フロー

### corpus.py の処理順序

1. **競技問題処理ループ**（L222-297）: `reasoning/*.txt` が存在し `problems.jsonl` に登録済みの問題を処理
2. **Downsample**: numeral(0.4)、gravity(0.6)、unit_conversion(0.6) を確率的に間引く
3. **人工データ処理ループ**（L299-347）: `augmentations/*.txt` を全件処理
4. **Duplication**（L349-387）: priority IDs のコピー追加 → カテゴリ別コピー追加
5. **corpus.jsonl に出力**（L389-414）: 全エントリをフラットに書き出し

競技問題と人工データは**セット化されず、独立サンプルとして同じ corpus に混在**する。

### train_sft.py の処理

```python
# 全 corpus を読み込み、included=True のものを全件学習
entries = load_corpus_entries()
entries = [e for e in entries if e["included"]]
examples = [TrainingExample.from_dict(e) for e in entries]

# カテゴリ均等分散バッチを作成
batches = _stratified_batches(examples, batch_size, rng)
```

`_stratified_batches()` はカテゴリごとにグループ化し、各バッチへ分散配置する。ただし、これはバッチ内のカテゴリ偏りを減らすためのものであり、**カテゴリルータ（異なる loss や optimizer 設定）ではない**。

### 学習時の `category` の役割

- **データ選別**: `choose_entry_to_include()` での採用判定
- **Downsample**: `DOWNSAMPLE_RATES` の参照
- **Duplication**: `DUPLICATE_COUNTS` の参照
- **バッチ分散**: `_stratified_batches()` での均等配置
- **ログ・集計**: `train_sft.py` L442-467 でカテゴリ別 loss・min logprob を集計・ログ

**`category` は token 化されておらず、モデルへの入力には含まれない。**

## 9. 推論時のデータフロー（カテゴリルータはない）

`notebook_tinker.py` の推論コード（L603-626）:

```python
user_content = (
    item.prompt
    + "\nPlease put your final answer inside `\\boxed{}`. For example: `\\boxed{your answer}`"
)
prompt = tokenizer.apply_chat_template(
    [{"role": "user", "content": user_content}],
    tokenize=False,
    add_generation_prompt=True,
    enable_thinking=True,
)
outputs = llm.generate(prompts, sampling_params=sampling_params, lora_request=...)
```

- モデルに渡すのは **`item.prompt` + boxed 指示のみ**
- **カテゴリ名は渡さない**
- モデルは prompt の文面・例示形式・記号パターンから、学習時に見た対応する出力パターンを再現する

「カテゴリを見て solver A/B/C を切り替える」仕組みは実装されておらず、**モデルが暗黙的に prompt の特徴から適切な出力様式を選択する**。

## 11. Tinker + SVD 変換問題（Training-Serving Misalignment）

### Tinker の位置づけ

Tinker は外部クラウド GPU リソースを API 経由で使う学習サービス。  
1回の学習に **約20分**（GPU 直接では 4 時間）。  
総学習コスト: 約 **$212**（Tinker クレジット）+ $60（Modal）+ $10（Kaggle/Colab）。

### Tinker adapter → Kaggle 提出形式の変換

Tinker が出力する adapter は Kaggle の提出形式と一致しないため変換が必要:

| 差分 | Tinker 形式 | 提出形式 |
|------|-----------|---------|
| キープレフィックス | `base_model.model.model` | `base_model.model.backbone` |
| Experts 重み | `experts.w1`, `experts.w2`（fused） | `experts.{i}.up_proj`, `experts.{i}.down_proj`（per-expert） |
| Mamba 層重み | `gate_proj` + `x_proj`（分離） | `in_proj`（統合） |
| lm_head | LoRA あり | 参照 adapter にはなし |

### SVD 変換の損失

`gate_proj + x_proj → in_proj` の変換に **SVD を使用**するが、**特異値質量の 75% しか捕捉できず lossy 変換** になる。  
この誤差により、学習時は正しく予測できていたトークンが推論時に誤生成される。

観察された症状:
- `\boxed` が書けない
- 直前トークンを繰り返す degenerate response
- テンプレートに従わず単語を飛ばす

→ README の記述からは、これは **主要因の一つ** と読めるが、**唯一の原因とまでは断定しない方が安全**

## 12. 参考: 既存アプローチ（スコア0.73）との比較

この節は **A-Open README / `nemotron` 実装の一次ソースではなく**、本リポジトリ側の既存実験と照らした**参考比較メモ**です。  
一次ソースから直接確認できる A-Open の特徴は次の点です。

- 学習データは **15,979 件**（競技 CoT + 人工補助データ）
- 人工補助データは **8,463 件**
- `priority_problem_ids.txt` による重点 oversampling と、`cryptarithm_deduce` の duplication がある
- `min logprob` を強く意識している
- `numeral` / `gravity` / `unit_conversion` は downsample している
- 弱点を一般推論としてではなく、**文字分解・括弧操作・bit 照合** といった token レベルの補助課題に分解している

したがって、A-Open の本質的な貢献は「**CoT を増やす**」だけでなく、**モデルの token レベル弱点を専用の補助データで重点訓練する**という発想にある、と整理するのが一次ソースに忠実です。

## 付録: README が指摘するモデルの弱点（引用）

> "Apparently, Nemotron is quite bad at spelling" — README L80（Cipher 節）

> "Apparently, Nemotron is also quite bad at splitting and concatenating symbols." — README L117（Cryptarithm 節）

> "Fix Nemotron's ability to convert text to characters. Even as I work with tokens, I do not control the tokenizer in submissions. The base Nemotron is quite bad at converting text to characters, and converting symbol groups to individual characters." — README L235（改善提案）

これらの弱点に対して、`spelling`, `splitting`, `concatenation` は **強い根拠をもって対策データと見てよい**。一方で `lstrip` は **同系統の補助データとみるのが安全** で、README からの直接根拠は相対的に弱い。

## 付録2: 著者による改善提案（README L230-237）

| 提案 | 内容 | 期待効果 |
|------|------|---------|
| **Misalignment 修正** | SVD 変換後に Tinker で post-finetune して損失を回復 | easy task の取りこぼし低減、training-serving gap の緩和 |

## 付録3: 反省と成功要因

### 成功要因

- **LeetCode 経験**による bit manipulation の CoT 設計（差の半分以上に寄与と著者が振り返る）
- **スカラー報酬を超えてデータを見る**：loss curve → カテゴリ別 loss → min logprob → 個別 token と段階的に掘り下げ
- **不要なアプローチを知っていた**：RL・蒸留が不要と判断できた
- **Tinker で高速反復**：4時間 → 20分のイテレーションサイクル
- **自作ダッシュボード**：学習状況を把握するカスタムフィルタリング UI を構築

### 反省

- **Numeral での失敗**：検証不足で簡単な問題を失点。misalignment の規模を事前に把握できていなかった
- **Held-out 検証セットがなかった**：リーダーボードスコアを事前予測できなかった
- **High-loss トークン調査の手動作業**：AI ツールにもっと任せるべきだった
