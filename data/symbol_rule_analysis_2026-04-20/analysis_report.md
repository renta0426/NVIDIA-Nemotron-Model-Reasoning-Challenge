## Executive Summary

- equation_numeric 系は『1 問 1 規則』ではなく、『1 問の中に 1〜3 個の演算子テンプレートが共存する』構造が強いです。
- cryptarithm 系は単純な文字列遊びではなく、8〜10 個前後の digit-like symbol と 2〜3 個の operator-like symbol から成る、隠し記号化算術として見るのが妥当です。
- 4 分類のままでは粗すぎます。少なくとも 直列連結型, 算術値型, 符号付き差分型, 桁別演算型, 未解決の複合型 に分けるのが実務的です。
- cryptarithm_guess は『ターゲット記号やターゲット演算子が局所文脈に存在しない』ため、局所プロンプトだけではなく train 全体の族情報を使う前提で設計すべきです。

### Symbolic Takeaway

- cryptarithm 系の大半は『出力が入力の部分列』ではありません。新記号が頻繁に出るため、純粋な削除・並べ替え問題ではなく、記号化された数値出力と見るべきです。
- 特に (総記号 13, 演算子 3, digit-like 10) が最多で、decimal-like な digit set と 3 演算子の組み合わせを強く示唆します。
- stable 署名 01234->0134 は『演算子削除』、01234->3401 は『右2文字 + 左2文字』です。これは numeric 系の cd+ab / operator-ignore 族と対応します。
- 一方で 01234->NN, 01234->NNN, 01234->2NN のような全新記号出力も多く、これは hidden digit system 上の加減乗除や剰余が結果側で新 digit を生む状況と整合します。

## Recommended Refinement

4 分類のままでは粗いので、少なくとも次の finer type へ分けるのがよいです。

- concat_swap_family: 右2桁+左2桁、または右2記号+左2記号。numeric の cd+ab、symbolic の 01234->3401 が対応。
- concat_drop_operator_family: 演算子を無視して残り4文字を返す族。symbolic の 01234->0134 が代表。
- arithmetic_value_family: x+y, x-y, x*y, x%y, y%x などの値そのものを返す族。symbolic では NN, NNN, NNNN が対応。
- signed_difference_family: op+(y-x), op+(x-y), (x-y)+op など、演算子を結果へ埋め込む族。
- digitwise_family: sum:ac_bd, diff:ac_bd, diff:ac_bd:strip0 のような桁別処理族。
- unresolved_composite_family: 上記 DSL でまだ拾えない複合規則。numeric_deduce に特に多いです。

## Bottom Line

- equation_numeric_guess / deduce は、演算子ごとにテンプレートを割り当てる DSL 問題です。
- cryptarithm_guess / deduce は、その DSL を数字ではなく記号化 digit system 上で出している問題です。
- guess 系は unseen operator / unseen symbol が混ざるため、prompt 単独の帰納よりも train 全体から族 priors を持ち込む設計が必要です。
- したがって solver を作るなら、まず numeric 系でテンプレート辞書を増やし、その後 cryptarithm 系へ symbol substitution 層を載せる順が最短です。
